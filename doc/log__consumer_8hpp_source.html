55  /*
56  * log_consumer : class intended as singleton for slave server
57  *
58  * Data members:
59  * - a pointer to slave stream (currently it also creates it, but future code should have a higher level
60  * object which aggregates both log_consumer and stream)
61  * - a queue of replication stream entry objects; the queue is protected by a mutex
62  * - m_apply_task_cv : condition variable used with m_queue_mutex to signal between consume daemon and
63  * dispatch daemon (when first adds a new stream entry in the queue)
64  * - m_is_stopped : flag to signal stopping of log_consumer; currently, the stopping is performed
65  * by destructor of log_consumer, but in future code, a higher level objects will handle stopping
66  * and destroy in separate steps;
67  * stopping process needs to wait for daemons and thread pool to stop; consume daemon needs to wait
68  * for stream to unblock from reading (so, we first signal the stop command to stream)
69  *
70  * Methods/daemons/threads:
71  * - a daemon which "consumes" replication stream entries : create a new replication_stream_entry object,
72  * prepares it (uses stream to receive and unpacks its header), and pushes to the queue
73  * - a dispatch daemon which extracts replication stream entry from queue and builds applier_worker_task
74  * objects; each applier_worker_task contains a list of stream_entries belonging to the same transaction;
75  * when a group commit special stream entry is encoutered by dispatch daemon, all gathered commited
76  * applier_worker_task are pushed to a worker thread pool (m_applier_workers_pool);
77  * the remainder of applier_worker_task objects (not having coommit), are copied to next cycle (until next
78  * group commit) : see dispatch_daemon_task::execute;
79  * - a thread pool for applying applier_worker_task; all replication stream entries are unpacked
80  * (the consumer daemon task is unpacking only the header) and then each replication object from a stream entry
81  * is applied
82  */
