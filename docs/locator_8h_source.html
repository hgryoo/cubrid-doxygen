130 /*
131  * Currently, classes does not have versions. So, fetching current, MVCC or
132  * dirty version lead to same result when classes are fetched. However,
133  * using LC_FETCH_CURRENT_VERSION is recommended in this case.
134  *
135  * When need to read an instance (DB_FETCH_READ) for SELECT purpose, use MVCC
136  * version type. This means visible version for current transaction, without
137  * locking. In this way, we respect the rule "do not lock instances at select"
138  * (reader does not block writer and writer does not block reader)
139  *
140  * When need to read an instance (DB_FETCH_READ) for UPDATE purpose, use DIRTY
141  * version type. In this case, the updatable version will be S-locked, if.
142  * exists. This guarantees that the object can't be deleted by concurrent
143  * transaction. If we don't lock the object, unexpected results may be obtained.
144  * That's because we will try to update later an object which we consider
145  * "alive", but was deleted meanwhile by concurrent transaction. Also,
146  * "..does_exists.." functions must use this version type (the only way to know
147  * if the object exists, is to lock it). Also, there are other particular
148  * commands like ";trigger" that need to use this version type (need locking).
149  *
150  * In some particular cases, you can use current version when read an instance
151  * (DB_FETCH_READ). For instance, if the object was already locked, you can
152  * use CURRENT version instead DIRTY (if the current transaction hold an lock
153  * on OID -> is the last version of the object -> no need to lock it again
154  * or to apply snapshot).
155  *
156  * When need to update an instance (DB_FETCH_WRITE), and the instance is not
157  * locked yet, use MVCC version type. In this case the visible version is
158  * searched into MVCC chain. Then, if exists, starting from visible version,
159  * updatable version is searched and X-locked. This is similar with
160  * update/delete executed on server side.
161  *
162  * ODKU use find unique (dirty version with S-lock). Since the object is
163  * locked, will be fetched using current or dirty version not MVCC version.
164  *
165  * In read committed, if have S, SIX, X or SCH-M lock on class, the instance
166  * must be fetched with current or dirty version. In RR, MVCC snapshot must
167  * be used in order to allow SERIALIZABLE conflicts checking.
168  *
169  * If the instance is not locked and its class doesn't have shared or
170  * exclusive mode, use MVCC version.
171  *
172  * au_get_new_auth uses IX-lock with dirty version when fetch all _db_auth
173  * instances. Each instance is fetched using S-lock and if an error occur, the
174  * instance is skipped.
175  *
176  * Currently, CUBRID tools use MVCC version when need to read instances.
177  *
178  */

181  LC_FETCH_CURRENT_VERSION = 0x01, /* fetch current version */

184  LC_FETCH_CURRENT_VERSION_NO_CHECK = 0x04, /* fetch current version and not check server side */
